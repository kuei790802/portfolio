從個人痛點到 AI Multi-Agent 實驗：這個專案的真實故事

起源：一個真實的個人痛點

我聽力幾乎滿分，閱讀偏弱。平時有閱讀英文新聞與社群的習慣，但實際考試時抓不到節奏。 多益閱讀練習題本「又厚又重，不方便攜帶」。我想要的是在通勤、午休、等飯這些碎片時段就能練的工具。

市面上現有 APP 大多是「學習導向」（單字、文法）或「聽力＋閱讀混合」，沒有純做閱讀衝刺、能精準控制時間的 APP。 不然就是訂閱費太貴。在 PTT 與 Dcard 也看到不少人有相同抱怨。

產品定位：把多益閱讀的方法論濃縮成 APP，讓使用者在 5/10/15/20 分鐘的時間限制下做題，模擬真實考試的時間壓迫感。

User Validation（小規模但真實）

5 位 dogfood 使用者（老婆、妹妹、3 位同事）

  • 一致回饋：「如果要準備多益會想用」
  • 5 人中 0 人提出「為什麼是這個時間區間」的疑問，時間設計命中需求
  • 我自己使用約 10 次，做了 ~80 題

已知 limitation（誠實標註）

  • n=5 不具統計意義
  • 都是身邊熟人，有 social desirability bias
  • 沒有做正式 user interview、沒有 JTBD 框架訪談
  • 沒有問「願意付多少錢」「會替代你現在的哪個工具」

Contribution Split角色聚焦於 spec / 架構 / review，工程實作交給 AI agent

我完成過資展國際 582 小時 Java 工程師養成班，具備 Spring Boot / React / SQL / RESTful API 等基礎技能。 但在 TOEIC Snack 中我有意識地將工程實作交給 AI agent 執行，自己聚焦於 spec、架構與審查層。 這不是能力限制，而是想驗證「AI 時代的 PM 角色：把問題定義到 Agent 能執行的程度」這個假設。

配套生態：另一個 AI Multi-Model 實驗

TOEIC_Question_Generator（獨立 GitHub repo）是專門服務於 TOEIC Snack 的 AI 出題管線。 使用多模型協作架構：高階模型（Sonnet）負責生成題目，低階模型（Haiku）負責驗證與核對， 通過後同步寫入 Supabase 資料庫。實際已產出數百題，本 APP 即從這個資料庫拉題目。

這個獨立 repo 的存在本身是個 PM 決策：把內容生產 pipeline 與產品 runtime 分離，避免 schema 與題庫管理耦合。 也是我第二個 multi-agent 設計（第一個是 Snack 本身的 CC × Gemini）。

目前狀態：MVP Hold

MVP 已完成核心練習迴圈、placement test、history、profile、settings、offline queue、error boundary。 經 dogfood 與 iOS 真機測試後，選擇停留在 MVP 階段，不立即送 App Store。

為什麼不上架？ 上架後實際營運要投入金錢（App Store 開發者帳號、Server cost、客服）與大量時間。 同期還在找工作、處理搬遷桃園、姊夫委託的 LINE bot 案、以及 PMP 考試準備。 這是一個有意識的「不上架」決策，而非半途而廢。

AI 協作流程 ★

這個專案最有價值的部分：我不小心摸出了「harness」框架的雛形
背景：開始這專案時（2026-01），社群還沒有大規模討論 AI Agent 的「harness」概念。 我為了管理 Claude Code 與 Gemini CLI 兩個 agent 之間的協作，設計了一套 HANDOFF 檔案協議 + STATUS.yml + loop control。 後來研究 harness 框架時，才發現自己摸索的方向與業界趨勢一致。 這套架構還很原始，但誤打誤撞踩到了前沿 AI 應用的關鍵設計問題。

HANDOFF 協議的設計動機

傳統 multi-agent 系統最大問題是狀態管理：兩個 agent 各自有 context，無法共享，重啟後遺失。我用文件系統作為共享狀態解決：

  • HANDOFF-CC.md：CC 寫完邏輯後，留下給 Gemini 的交接單（含 wireframe 路徑、hooks 介面、注意事項）
  • HANDOFF-GEMINI.md：Gemini 完成 UI 後，回報結果給 CC
  • STATUS.yml：兩個 agent 共讀的進度狀態（current_task、progress、blockers、loop_control）
  • plan.md：TDD checklist，當作 source of truth
  • .handoffs/YYYY-MM-DD-{agent}-{topic}.md：完成後歸檔，可追溯
關鍵洞察：把 agent 間的協作具現化為檔案，不只解決狀態問題，還順便解決： （1）可審計：所有決策有紙本，（2）可中斷可恢復：session crash 後重讀檔案即恢復脈絡，（3）人類可介入：我隨時可以手動編輯 HANDOFF 介入流程。

Loop Control 與 Auto Mode

STATUS.yml 有一個 loop_control: continue | stop 旗標。 當 CC 被用 -p 旗標（非互動模式）呼叫時，會檢查這個值決定是否寫下一份 HANDOFF-CC.md 讓閉環繼續。 搭配 frontend/scripts/orchestrator 啟動的 iTerm2 3-pane dashboard， 可以實現自主開發循環：CC 寫 → Gemini 實作 → Gemini 提交 → CC 接下一輪。

I-3 的單日 +51 tests 與 I-5 的同日完成 Phase 6，就是這個 auto-loop 跑出來的。

真實的 AI 失敗案例 （不藏拙）

QA-011：Gemini CLI worker hung 12.9 hours
QA 階段發現 Part 6/7 長文閱讀題的選項會被隱藏在 fold 之下，使用者看不到。 我把這個 bug 開了 handoff 給 Gemini CLI 修，Gemini 卡了 12.9 小時沒有任何輸出與檔案變更，最後被 Kermes（另一個 QA agent）終止。

恢復策略：QA agent 直接接手實作了 frontend guard（未答題時 disable Next/Submit + 提示捲動）， 並補上 PracticeScreen.test.tsx 的 regression coverage。最終 iOS Simulator 視覺 retest 通過。

PM 學到的教訓：multi-agent 系統需要設計 fail-over 機制。 不能假設每個 agent 都會成功，要有 timeout、有 escalation path、有 fallback executor。

其他 AI 出包的時刻還有約 10 次（記憶模糊），常見於： （a）spec 描述不夠精準導致 agent 走偏， （b）跨檔案的 implicit dependency 沒在 HANDOFF 註明， （c）測試的 mock 與真實 API 行為偏離（QA-002 的根因之一）。

這是成本最佳化的 multi-model 設計：用便宜模型把關品質，用昂貴模型做創意產出。 目前已產出數百題進入 production 資料庫。

這個架構對 PM 工作的啟示

  • Spec 是新 Code：當 AI 能執行任何明確需求，PM 的競爭力就在「能不能寫出可被執行的 spec」
  • 檔案是新 API：agent 之間用檔案溝通比用 message passing 穩定（可中斷可恢復）
  • HANDOFF 是新 Sprint Review：每次 handoff 就是一次小型 review，迫使把工作明確化
  • Loop Control 是新 Backlog Grooming：用 continue / stop 旗標決定下一輪做什麼，比每天開會更輕量
  • Audit Trail 內建：.handoffs/ 目錄等於 sprint history，未來給 stakeholder 看一目了然

反思

3 個 PM Lessons · What I'd Do Differently · 未來方向
誠實前置：這專案一開始不是以 PM 視角執行，而是「測試 multi-agent 開發」的工程實驗。 它能變成 PM portfolio，是因為事後我用 PM 的角度反思了它的缺失與啟發。 這份 Reflections 是 portfolio 中對自己最誠實的一節。

3 個 PM Lessons

Lesson 1：PM 的價值是「描述清楚需要被執行的事」

我有 Java 全端工程訓練基礎，但在這個專案選擇將工程實作交給 AI agent 執行，自己聚焦於 80% 的規格、60% 的架構決策、50% 的 prompt 設計。 AI 把「執行」變成 commodity 之後，限制 throughput 的就只剩 spec quality。 Spec 寫得清楚 → AI 線性放大；spec 模糊 → 不管多少 agent 都救不了。

傳統 PM 訓練重視「跨團隊溝通」「stakeholder management」，這些依然重要。 但 AI 時代多了一個技能：把需求精確到 agent 可以執行。 這需要工程理解、邊界條件思維、測試 case 想像力，這也是為什麼這份 portfolio 能用 TDD checklist 當作 spec 載體。

證據：QA-011 案例顯示，Gemini 卡 12.9 小時的根因之一是 HANDOFF spec 對 "scroll-to-options" 的描述不夠具體。 後來我直接補了「未答題時 disable Next/Submit + 顯示提示」這個具體 acceptance criteria，問題才解決。

Lesson 2：Velocity 是 vanity metric，Outcome 才是 KPI

13 天跑完 160 tests / 6 phases 的速度，遮蔽了一個事實： MVP 完成 2 個月後，真機 dogfood 才發現 Guest Submit 從未真的能用（QA-002）。 Phase 5 的核心 acquisition flow 在我宣稱 "completed" 時是壞的。

根因：Definition of Done 只定義為「測試通過 + TS 0 errors」，沒有包含「user-validated」。 單元測試 mock 掉了 JwtAuthGuard，所以 guest submit 在測試環境永遠通過，但在真實 API 環境是 401。

下一輪會做的事：每個 phase 的 exit criteria 必須包含至少一次 end-to-end real-device dogfood pass。 每個 user-facing 功能必須對應一個 outcome metric（DAU、completion rate、retention），不只 output metric（test count）。

證據：STATUS.yml 標 phase6-launch-prep completed at 2026-03-08，但 QA-002 在 2026-05-07 才被發現並 close。

Lesson 3：Build-first vs Plan-first 不是非黑即白

ROADMAP.md 是 MVP 完成後才定稿的（2026-03-06，commit hash 顯示）。 傳統 PMI 視角這是 anti-pattern，認為「應該先 plan 再 execute」。 我在 PMP / PSM I 訓練中也被教這套。

但這專案的實際經驗讓我重新思考：對 solo + AI 的 lean prototype， build-first 可能更誠實，先建一個能跑的 MVP 來驗證 spec 與工具鏈，再用 retrospective roadmap 規劃下一輪。 理由：（a）solo 沒有跨團隊溝通成本，plan 的邊際效益低， （b）AI 工具行為仍在快速演化，過度提前規劃容易過時， （c）prototype 本身就是最便宜的 prototype，比 plan 更能揭露未知。

學到的不是「以後一定 plan first」，而是分清楚場景：

  • 必須 plan-first：跨團隊、高風險、有 compliance、有金錢承諾、user-facing launch
  • 可以 build-first：個人原型、技術未知、unit cost 低、可拋棄
證據：ROADMAP.md "Last updated: 2026-03-06"，但 git log 顯示 Auth / Placement / Core Practice 在 2026-02-24 ~ 02-28 就完成了。

Lesson 4：不做什麼比做什麼更需要紀律

TOEIC Snack 跑完 6 個 phase / 172 個測試，但 MVP 至今沒有上架。 事後反省：很多東西其實不該做進 MVP，例如完整的 OAuth 三入口、achievements 5 種徽章、notifications 排程。 這些都增加了複雜度，但對驗證「碎片時間練習有沒有市場」這個核心假設毫無貢獻。

「不做什麼」需要更強的紀律，因為「做」是 default、「不做」要主動承擔取捨。 PM 最常見的錯誤是「什麼都想做」，把 backlog 塞滿後再來「優先序排序」，但這已經輸了， 因為團隊的精力與情緒成本是不可逆的。

下一個專案（保險 Bot）我嚴格實踐了這條：明確列出 Non-Goals（壽險、客戶端自助、自動發訊息、多租戶、Email 分流）放進 PRD， Phase 1 砍掉「紙本續保單追蹤」整套流程，因為訪談發現根本沒人在追。 範圍縮減反而讓 Pilot 在 4/30 截止前能聚焦完成。

證據：保險 Bot ADR-003「Phase 1 不做紙本追蹤流程」明確紀錄此判斷，PRD 中的 Non-Goals 區塊與目標區塊同等重要。

What I'd Do Differently（如果重做一次）

  1. Harness First, MVP Second
    先把 multi-agent harness 架構完整建好（loop control、fail-over、自動 escalation、metrics dashboard）， 再讓 agent 從規格自動跑到 MVP。我全程不介入，等收到 MVP 再 review。 這次是邊做邊架 harness，浪費很多重工。
  2. Discovery 補做（已在保險 Bot 落地）
    TOEIC Snack 的 Discovery 規模偏小（5 人 dogfood + 自身使用驗證 + 論壇觀察），未做正式 user interview。 運氣好命中需求，但這不可複製。下一個專案（保險 Bot）我用《The Mom Test》紀律補上： 正式訪談 3 位從業人員、區分直接證據與推論的分級框架、訪談結果直接推翻初版 PRD 並校準為 v2.0。 Discovery 不能跳，但可以分階段補做。
  3. Outcome Metrics Day 0 就接
    即使沒有正式 analytics，至少自己手動追：每天用了幾次、做了幾題、答對率、平均 session 長度、有無 crash。 這次是 0 metric 跑了 3.5 個月才在 dogfood 發現一堆問題。
  4. Definition of Done 必含 user-validated
    每個 phase 完成要包含至少 1 次 real-device dogfood pass，不只測試綠燈。
  5. Risk Register 從 Day 0 維護
    不是事後重建（這次是事後重建）。每週更新一次。

未來方向

本 APP：先停在 MVP 階段，視就業狀況、市場機會、姊夫案的進度，再決定是否上架。不上架 ≠ 失敗，這是有意識的 capacity allocation 決策。

Multi-Agent Workflow：這套 HANDOFF 協議的概念，我打算在下個專案（不論 side project 或工作場景）繼續演化。可能會嘗試：

  • 引入 timeout + escalation（解 QA-011 那種 Gemini hung 問題）
  • 把 HANDOFF 從 markdown 升級為 structured schema（YAML / JSON Schema）
  • 加上 cost tracking（每個 iteration 的 token 用量與成本）
  • 嘗試 3+ agent 協作（加入 reviewer / security 角色）

PM 職涯：希望第一個 PM 工作落在軟體、電信、智慧交通或 AI 應用導向的軟體公司。可以把這次的 multi-agent 經驗延伸到實際產品團隊，尤其在「怎麼用 AI 加速軟體交付而不犧牲品質」這個題目上。
