從個人痛點到 多 Agent AI 實驗：這個專案的真實故事

問題探索：從個人痛點到二手驗證

起點是真實的個人痛點：我聽力幾乎滿分、閱讀偏弱，平時有閱讀英文新聞與社群的習慣，但實際考試時抓不到節奏。 多益閱讀練習題本又厚又重不方便攜帶，我想要的是在通勤、午休、等飯這些碎片時段就能練的工具。

接著做二手研究驗證痛點是否普遍：我到 PTT 與 Dcard 爬文，確認有不少考生跟我同樣狀況（聽力分數高、閱讀缺技巧、抓不到時間節奏）。 這讓我確認這不是 n=1 的個人問題，而是一個值得解的群體需求。

競品掃描找出市場空缺：現有 APP 大多是「學習導向」（單字、文法）或「聽力＋閱讀混合」， 沒有純做閱讀衝刺、能精準控制時間的工具，付費門檻也偏高。

產品定位：把多益閱讀的方法論濃縮成 APP，讓使用者在 5/10/15/20 分鐘的時間限制下做題，模擬真實考試的時間壓迫感。

配套生態：另一個 AI Multi-Model 實驗

TOEIC_Question_Generator（獨立 GitHub repo）是專門服務於 TOEIC Snack 的 AI 出題管線。 使用多模型協作架構：高階模型（Sonnet）負責生成題目，低階模型（Haiku）負責驗證與核對， 通過後同步寫入 Supabase 資料庫。實際已產出數百題，本 APP 即從這個資料庫拉題目。

這個獨立 repo 的存在本身是個 PM 決策：把內容生產 pipeline 與產品 runtime 分離，避免 schema 與題庫管理耦合。 也是我第二個多 Agent 設計（第一個是 Snack 本身的 CC × Gemini）。

AI 協作流程 ★

這個專案最有價值的部分：我不小心摸出了「harness」框架的雛形
背景：開始這專案時（2026-01），社群還沒有大規模討論 AI Agent 的「harness」概念。 我為了管理 Claude Code 與 Gemini CLI 兩個 agent 之間的協作，設計了一套交接檔案協議（HANDOFF） + STATUS.yml 進度檔與迴圈控制。 後來研究 harness 框架時，才發現自己摸索的方向與業界趨勢一致。 這套架構還很原始，但誤打誤撞踩到了前沿 AI 應用的關鍵設計問題。

交接協議的設計動機

傳統多 Agent 系統最大問題是狀態管理：兩個 agent 各自有 context，無法共享，重啟後遺失。我用文件系統作為共享狀態解決：

  • HANDOFF-CC.md：CC 寫完邏輯後，留下給 Gemini 的交接單（含 wireframe 路徑、hooks 介面、注意事項）
  • HANDOFF-GEMINI.md：Gemini 完成 UI 後，回報結果給 CC
  • STATUS.yml：兩個 agent 共讀的進度狀態（current_task、progress、blockers、loop_control）
  • plan.md：TDD checklist，當作 source of truth
  • .handoffs/YYYY-MM-DD-{agent}-{topic}.md：完成後歸檔，可追溯
關鍵洞察：把 agent 間的協作具現化為檔案，不只解決狀態問題，還順便解決： （1）可審計：所有決策有紙本，（2）可中斷可恢復：session crash 後重讀檔案即恢復脈絡，（3）人類可介入：我隨時可以手動編輯交接檔案介入流程。

Loop Control 與 Auto Mode

STATUS.yml 有一個 loop_control: continue | stop 旗標。 當 CC 被用 -p 旗標（非互動模式）呼叫時，會檢查這個值決定是否寫下一份 HANDOFF-CC.md 讓閉環繼續。 搭配 frontend/scripts/orchestrator 啟動的 iTerm2 3-pane dashboard， 可以實現自主開發循環：CC 寫 → Gemini 實作 → Gemini 提交 → CC 接下一輪。

I-3 的單日 +51 tests 與 I-5 的同日完成 Phase 6，就是這個 auto-loop 跑出來的。

容錯機制：當 AI agent 卡住時怎麼辦

實際發生的狀況：QA 階段發現 Part 6/7 長文閱讀題的選項會被隱藏在畫面下方，使用者看不到。 我把這個修正交給其中一個 agent 處理，但它長時間沒有產出。

我設計的應對：因為工作流本來就有進度監控與接手機制，停滯被即時偵測， 改由備援的執行流程接手，實作了防呆 guard（未作答時鎖定下一題與送出、提示使用者捲動）， 並補上對應的 regression 測試，最終真機驗證通過。

PM 觀點：多 Agent 系統的關鍵不是假設每個 agent 都會成功，而是預先設計好監控、逾時與接手機制。 這跟管理人類團隊一樣：要有進度可視性與備援，不能把單點風險留在流程裡。

過程中也累積了讓 agent 更穩定的做法：把規格與驗收條件寫精準、在交接文件標明跨檔案相依、讓測試貼近真實 API 行為。這些都回饋進了我的協作流程設計。

這個架構對 PM 工作的啟示

  • Spec 是新 Code：當 AI 能執行任何明確需求，PM 的競爭力就在「能不能寫出可被執行的 spec」
  • 檔案是新 API：agent 之間用檔案溝通比用 message passing 穩定（可中斷可恢復）
  • 交接是新的 Sprint Review：每次 handoff 就是一次小型 review，迫使把工作明確化
  • Loop Control 是新 Backlog Grooming：用 continue / stop 旗標決定下一輪做什麼，比每天開會更輕量
  • Audit Trail 內建：.handoffs/ 目錄等於 sprint history，未來給 stakeholder 看一目了然

反思

4 個 PM 心得、下一步強化方向、未來規劃
這個專案起點是「測試多 Agent 開發」的技術實驗，我用 PM 的視角把它走完整、並萃取出可複用的方法論。 以下是我從中固化下來的判斷原則，其中多項已落實到後續專案。

3 個 PM Lessons

Lesson 1：PM 的價值是「描述清楚需要被執行的事」

我有 Java 全端工程訓練基礎，但在這個專案選擇將工程實作交給 AI agent 執行，自己聚焦於 80% 的規格、60% 的架構決策、50% 的 prompt 設計。 AI 把「執行」變得低成本之後，限制產出效率的就只剩 spec quality。 Spec 寫得清楚 → AI 線性放大；spec 模糊 → 不管多少 agent 都救不了。

傳統 PM 訓練重視「跨團隊溝通」「stakeholder management」，這些依然重要。 但 AI 時代多了一個技能：把需求精確到 agent 可以執行。 這需要工程理解、邊界條件思維、測試 case 想像力，這也是為什麼這份 portfolio 能用 TDD checklist 當作 spec 載體。

證據：QA-011 案例顯示，Gemini 卡 12.9 小時的根因之一是 交接文件的規格對 "scroll-to-options" 的描述不夠具體。 後來我直接補了「未答題時 disable Next/Submit + 顯示提示」這個具體 acceptance criteria，問題才解決。

Lesson 2：速度是過程指標，使用者驗證才是完成標準

13 天跑完 6 個 phase、172 個測試，速度很可觀。但我清楚「測試全綠」與「使用者實際走得通」是兩件事， 所以在測試之外安排了 iOS 真機試用，主動驗證單元測試覆蓋不到的整合情境。

這個動作有價值：我因此發現並修復了 guest 提交流程的一個整合問題（QA-002），完成部署與真機複測。 如果只看測試數字，這個問題會被帶到上架後才爆發。

帶走的原則：每個面向使用者的功能，完成定義都要包含至少一次真機端到端驗證， 並對應一個結果指標（如完成率、留存），而不只是產出指標（測試數）。這個原則我已落實到後續專案。

Lesson 3：方法論要看情境選，不是非黑即白

本專案我刻意採 build-first：先快速做出可運作的 MVP 驗證規格與工具鏈，再回頭定下完整 Roadmap。 我有 PMP / PSM I 的 plan-first 訓練，但判斷對 solo + AI 的原型驗證，build-first 更有效率： 沒有跨團隊溝通成本、AI 工具仍在演化、可運作的原型比計畫書更能揭露未知。

重點不是「哪個方法比較好」，而是能依情境選對方法：

  • 採 plan-first：跨團隊、高風險、有法規或合約承諾、正式對外發布
  • 採 build-first：個人原型、技術未知、成本低、可快速驗證

能清楚說出「為什麼這個專案這樣選」，比死守單一方法論更接近真實的 PM 判斷。

Lesson 4：不做什麼比做什麼更需要紀律

TOEIC Snack 跑完 6 個 phase / 172 個測試，但 MVP 至今沒有上架。 事後反省：很多東西其實不該做進 MVP，例如完整的 OAuth 三入口、achievements 5 種徽章、notifications 排程。 這些都增加了複雜度，但對驗證「碎片時間練習有沒有市場」這個核心假設毫無貢獻。

「不做什麼」需要更強的紀律，因為「做」是 default、「不做」要主動承擔取捨。 PM 最常見的錯誤是「什麼都想做」，把待辦清單塞滿後再來「優先序排序」，但這已經輸了， 因為團隊的精力與情緒成本是不可逆的。

下一個專案（保險 Bot）我嚴格實踐了這條：明確列出 Non-Goals（壽險、客戶端自助、自動發訊息、多租戶、Email 分流）放進 PRD， Phase 1 砍掉「紙本續保單追蹤」整套流程，因為訪談發現根本沒人在追。 範圍縮減反而讓 Pilot 在 4/30 截止前能聚焦完成。

證據：保險 Bot ADR-003「Phase 1 不做紙本追蹤流程」明確紀錄此判斷，PRD 中的 Non-Goals 區塊與目標區塊同等重要。

下一步會強化的方向

這個專案讓我看到自己工作流可以再升級的地方，其中幾項已經在後續專案（保險 Bot）落地。

  1. 協作框架先行
    把多 Agent 工作流的監控、逾時、自動接手與成本追蹤先建置完整，再讓 agent 從規格自動跑到 MVP， 把人的介入集中在規格與審查。這次邊做邊建已驗證可行，下一版可以更系統化。
  2. Discovery 前置（已在保險 Bot 落地）
    本專案已做輕量 Discovery（個人痛點、論壇二手研究、競品掃描、5 人試用），方向驗證足夠啟動 MVP。 下一個專案我把正式用戶訪談前置：用《The Mom Test》紀律訪談 3 位從業人員、建立證據分級， 訪談結果直接校準了 PRD。這個升級已經完成並見效。
  3. 結果指標從第一天就接
    在開發初期就接上使用行為追蹤（使用頻率、完成率、答對率、平均時長、穩定度）， 讓產品決策有數據支撐，而非只靠產出指標。
  4. 完成定義納入真機驗證
    每個面向使用者的功能，完成都包含至少一次真機端到端驗證。這個原則已落實。
  5. 風險清單從第一天維護
    把風險盤點變成每週更新的常態文件，而非階段性回顧。

未來方向

本 APP：先停在 MVP 階段，視就業狀況、市場機會與其他專案的進度再決定是否上架。停在 MVP 是基於資源配置的主動取捨，不是中斷。

多 Agent 工作流：這套交接協議的概念，我打算在下個專案（不論 side project 或工作場景）繼續演化。可能會嘗試：

  • 把 agent 逾時與自動接手機制做得更系統化
  • 把交接協議從 markdown 升級為 structured schema（YAML / JSON Schema）
  • 加上 cost tracking（每個 iteration 的 token 用量與成本）
  • 嘗試 3+ agent 協作（加入 reviewer / security 角色）

PM 職涯：希望第一個 PM 工作落在軟體、電信、智慧交通或 AI 應用導向的軟體公司。可以把這次的多 Agent 經驗延伸到實際產品團隊，尤其在「怎麼用 AI 加速軟體交付而不犧牲品質」這個題目上。
