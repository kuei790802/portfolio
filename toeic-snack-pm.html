TOEIC Snack PM Case Study

AI 協作產品實驗｜從個人痛點到 MVP 交付與產品取捨
一個多益閱讀練習 App，如何從「碎片時間練閱讀」的個人痛點出發，收斂成限時練習 MVP， 再用實機試用、iOS QA、風險 register 與 AI agent 交付機制驗證產品假設。 這份 case 的重點不是只做出 App，而是如何定義問題、控管交付品質，並判斷何時不該急著上架。
期間：3.5 個月 / 約 25 天實作
角色：Solo PM / AI Agent Orchestrator
交付：13 screens · 172 tests · iOS QA
狀態：MVP demo 可用，商業驗證前暫緩上架

Case Brief

這個案例展現的是 PM 視角的產品取捨，不只是技術交付。

我從個人痛點出發收斂出 MVP 範圍，用實機 QA、handoff 文件與風險清單控管交付品質，並在商業假設未驗證前主動暫緩上架，把資源留給更高優先序的驗證。

  • 用實機 QA、handoff、風險清單控管交付品質

問題與假設

把個人痛點轉成可驗證的產品問題，而不是直接開始做功能

起點：我自己的痛點，但不把 n=1 當市場

我聽力接近滿分，閱讀相對弱。平常會讀英文新聞與社群，但考試時容易抓不到時間節奏。 傳統題本厚重，不適合通勤、午休、等飯這些零碎時段。

這是一個真實痛點，但一開始只是 n=1 的個人經驗。 因此我先做二手研究與競品掃描，確認問題是否有機會擴大。

二手研究

從 PTT、Dcard 觀察到部分考生有類似情境：聽力高於閱讀、閱讀做不完、需要時間節奏訓練。

競品掃描

多數 App 偏單字、文法或綜合練習，較少聚焦「限時閱讀衝刺」與碎片時間使用。

核心產品假設

假設我如何轉成 MVP 功能當時證據
使用者需要短時間練閱讀 提供 5/10/15/20 分鐘練習入口，而不是完整模考。 個人痛點 + 論壇二手研究。
時間壓迫是閱讀弱點的一部分 練習流程內建 timer 與 time bank，讓使用者感受到考試節奏。 自身考試經驗 + 競品空缺。
難度需要被分層 設計 placement test 與 Green / Blue / Gold 分級。 產品直覺，尚缺正式驗證。
題庫可以用 AI pipeline 補足 另建 TOEIC_Question_Generator，將內容生產與 App runtime 分離。 技術可行，但題目品質仍需人工與使用者驗證。
分階段驗證：這些假設足以支持我先做 MVP 與小規模試用；若要上架與商業化，下一階段會補上付費意願、留存與題庫品質驗證。 這也是我選擇先完成 MVP、而不是急著上架的主因。

驗證策略

先用低成本方式確認方向，再把更重的驗證留給下一階段

我實際做過的驗證

驗證方式得到什麼證據強度我的判斷
個人使用 自己使用約 10 次，做約 80 題，確認核心流程能支援碎片時間練習。 早期 確認核心流程可用，作為後續外部驗證的基礎。
論壇二手研究 確認閱讀速度、時間配置、閱讀做不完是常見討論。 方向性 支持問題存在，下一步再驗證解法採用意願。
5 位家人同事試用 一致回饋「如果準備多益會想用」，未質疑時間區間設計。 方向性 確認使用情境有共鳴，適合作為 MVP 往下一階段推進的依據。
iOS QA 找出 11 個問題，修復 4 個；其中 guest submit 問題已完成部署與複測。 功能驗證 證明測試全綠不等於使用者走得通，完成定義要包含端到端驗證。

QA / 風險判斷

不是只有「測試通過」，而是確認使用者流程真的走得完

QA report 把技術綠燈和產品可上架拆開看：自動化測試可以通過，但 mobile guest flow、placement、screen coverage 和 production config 仍會影響是否能交給外部使用者。

Launch Gate 仍不急著上架 核心 demo 已可用，但 placement、screen coverage、production trust 仍需補齊。

下一階段要補強的驗證

付費意願

規劃用價格測試、登錄頁轉換率、訪談或預購，確認使用者是否願意付費。

留存

接上 DAU、留存、平均使用時長、連續使用天數等成果指標。

題庫品質

AI 產題已驗證可行，下一步會系統化檢查題目準確性、難度分布與解析品質。

營運成本

上架前補 App Store、server、題庫維護、客服與錯題回報流程的成本試算。

暫緩上架是產品時程的主動取捨： MVP 已能展示核心流程，但在商業假設完成驗證前，我選擇先保留彈性，把資源留給更高優先序的驗證與求職作品整理。

範圍取捨

完成 MVP 後，我重新檢視哪些功能真正服務核心假設
完成後的取捨調整： OAuth、徽章、通知、離線佇列等功能讓 MVP 更完整，也讓我更清楚看到一件事： 早期產品最重要的不是完整度，而是每個功能是否直接服務核心假設。

如果重做一次，我會這樣切 MVP

功能原本做法現在的 PM 判斷
限時閱讀練習 已做 5/10/15/20 分鐘練習。 保留 這是核心假設。
題目與解析 AI 題庫 pipeline + Supabase 題庫。 保留但縮小 先做少量高品質題目與人工驗證。
Placement test 15 題分級流程。 簡化 初期可改成自選難度，避免 onboarding 太重。
OAuth 三入口 Google / Apple / Email / Guest。 延後 早期用 guest 或 email magic link 即可。
徽章與通知 做 achievements 與 notification scheduling。 延後 留存驗證前，先不要過早做互動黏著包裝。
離線佇列 做 offline queue 與 retry。 視情境 若先做 TestFlight 小樣本，可以暫時不做完整離線策略。

我後來怎麼修正這個工作習慣

在後續保險續保 LINE Bot 專案，我把這個教訓轉成具體做法： PRD 一開始就列出不做事項，並把不做事項視為和功能需求同等重要。 例如壽險、多租戶、客戶端自助、Email 分流、紙本續保追蹤都先排除，讓第一階段聚焦在最能降低人工作業負擔的流程。

PM 能力不是把清單排滿，而是讓團隊聚焦在最重要的事。 多做功能通常比較容易被看見，但真正成熟的產品判斷，是能把資源集中在最能推進驗證的地方。

AI Contribution Split：我刻意把工程工作交出去

我有 Java / Spring Boot / React / SQL / RESTful API 基礎，這個專案刻意把工程實作整段交給 AI agent。 這不是能力空白，而是 PM 取捨：我把時間集中在規格、架構與品質卡控，讓「PM 把問題拆到 agent 可執行」這個假設被真正驗證一次。

我設計的交接機制

AI 協作閉環：把 agent 產出變成可審查交付

這套流程不是「請 AI 寫 code」，而是用規格、交接文件、測試與 QA 證據把任務切成可追溯的最小交付單元。

29 份歷史交接 + 最新 closeout
PM / Spec 定義問題與驗收條件 用 PRD、plan.md、wireframe 與測試清單，先界定 agent 應該交付什麼。
Claude Code 拆邏輯、型別與測試 負責 hooks、stores、services、interfaces 與邏輯測試，產出 HANDOFF-CC。
Gemini 接手 UI 與 Git 流程 依設計系統與 handoff 範圍實作畫面，執行測試、commit、merge、push。
Human Review QA、複測、決定是否關閉 用真機 QA、測試結果與部署狀態決定任務完成、回補文件或重新開 issue。

配套管線：多模型 AI 出題

App 的題目來自另一個獨立 repo TOEIC_Question_Generator。 這是我第二個多 agent 設計：用 Sonnet 生成題目，再用 Haiku 做驗證與核對，通過後同步寫進 Supabase，已實際產出數百題。

為什麼分兩個模型

生成模型負責創造力，驗證模型負責規則與一致性。讓便宜模型做檢查，可以把單題成本壓低，並把錯誤攔在寫進資料庫之前。

為什麼拆成獨立 repo

內容生產 pipeline 與 App runtime 解耦，schema 變動、模型替換、出題策略調整都不會波及 App 部署。這是刻意的 PM 架構決策。

下一步與聯絡

如果這份產品要繼續推進，下一階段會做的事

MVP 已展示核心練習迴圈與 AI 出題管線的可行性。 下一階段的優先序不是再加功能，而是把付費意願、留存指標、題庫品質、外部用戶試用這四件事的證據補齊，把資源留給更高優先序的工作。
